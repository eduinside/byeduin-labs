- 원인: 게스트의 close 이벤트 핸들러가 저장된 'paused' 상태를 'active'로 덮어쓰는 race condition
- 해결: 연결을 닫기 전에 STATE.isClosing = true 플래그 설정 (line 2602, 2643)
- close 핸들러가 STATE.isClosing 체크하여 저장 로직을 건너뜀
- **line**: 2599-2605 (pause), 2640-2646 (complete)

### 8. ✅ 게스트 완전 종료 메시지 표시 버그
**상태**: COMPLETE (commit e00de60 포함)
- 호스트가 '완전 종료' 버튼 클릭 시 게스트가 잘못된 '일시 중단' 메시지를 받는 문제
- 원인: 게스트가 'end' 메시지를 받고 STATE.role을 null로 리셋한 후, conn.on('close') 핸들러 실행 시
  - 조건문 `if (STATE.role === 'guest' && !STATE.isClosing)`이 false가 됨 (STATE.role이 null)
  - else 블록에서 `endChatSession()`이 호출되어 다시 `_endChatSession('pause')`가 실행됨
- 해결:
  1. close 핸들러에서 disconnect 토스트를 `if (!STATE.isClosing)` 조건으로 제한 (line 3022-3025)
  2. `endChatSession()` 호출을 `else if (!STATE.isClosing)` 조건으로 변경 (line 3100-3102)
- 이제 게스트가 호스트의 완전 종료 시 올바른 "대화가 완전 종료되었습니다" 메시지만 표시
- **line**: 3022-3025, 3100-3102

## 📝 개발 노트

### 알려진 이슈 및 제한사항
1. **localStorage 5-10MB 제한** (현재 상태: ✅ 안전)
   - **현재 사용량**: 0.3% (13 KB / 5 MB 기준)
   - **최대 20개 세션 저장**: 메시지 0~100개 기준 ~2 MB
   - **한계 도달**: 메시지 1,000개/세션 × 20개 = ~10 MB (초과)
   - **결론**: 통상적 사용에서는 문제 없음
   - **장기 해결**: IndexedDB 마이그레이션 (필요 시 2-3시간 소요)
   - **참고**: 코드에서 자동으로 20개 세션 제한하고 있음 (line 1912)

2. **PeerJS 신뢰성**
   - STUN/TURN 서버 선택에 따라 연결성 달라짐
   - NAT/방화벽 환경에서 일부 네트워크 접속 불가 가능

3. **메시지 동기화**
   - 게스트 재입장 시 이전 메시지 미동기
   - 향후 개선: 중앙 메시지 서버 도입

### 기술적 고려사항
- **PeerJS Peer ID**: 동일 탭에서 같은 ID 사용 불가
- **메시지 relay**: 호스트만 게스트 메시지를 다른 게스트에게 전달
- **DM 대상**: 게스트는 호스트에게만 DM 가능 (호스트는 모든 게스트에게 DM 가능)

### 성능 지표
- 메시지 렌더링 1000개 이상 시 느려질 수 있음 (가상 스크롤 필요)
- 파일 크기: index.html ~2800 lines
- 번들 크기: PeerJS (1.5.2) + QRCode.js 포함 ~200KB
